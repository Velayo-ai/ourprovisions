Preflight Before you build

Orient yourself and confirm you're starting from solid, known ground.

Confirm which branch you're on and that it's clean
You should almost always start on dev with a clean tree. Uncommitted changes from last time are a red flag — deal with them before building.
git branch
git status
Pull latest so you're not building on a stale local copy
git pull origin dev
Read the top of SESSION_LOG.md and the current ROADMAP NOW items
The last session's "Next session" block tells you the goal and the "Done when." Don't reinvent it.

Decide: is this a DESIGN session (chat) or a BUILD session (Claude Code)?
Design/architecture/decisions → do it in chat, produce a handoff file. Implementation → Claude Code. If both, design first, then hand off.

In-flight While you build

The discipline reminders — these are the rules that, when skipped, cost you whole sessions.

Commit before switching branches — every time
Lost work before from branch-switching with uncommitted changes. One step at a time, verify between steps.
After any deploy, hard-refresh ALL clients before judging (Ctrl+Shift+R)
Stale JS produces false negatives — a fix that looks broken is often just an old bundle. Don't trust a soft refresh.
Keep Claude Code prompts surgical — single file, exact matches
Grep for the target, view the exact line range, then edit. Large multi-file prompts fail more often.
For Supabase function changes, verify the save actually took
The SQL Editor has silently kept old versions. Confirm with the function definition.
select pg_get_functiondef(oid) from pg_proc
where proname = '<function_name>';
Watch for debug console.log lines — note them to strip before merge
Debug logging is fine on dev; it must not ride to main. Leave yourself a note in the commit if any remain.
